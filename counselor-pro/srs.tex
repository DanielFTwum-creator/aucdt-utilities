\section{Introduction}

\subsection{Purpose}
This document defines the functional and non-functional requirements for the CounselorPro application. It serves as a foundational agreement between the development team, stakeholders, and users.

\subsection{Document Conventions}
This document follows the IEEE 830-1998 standard. All requirements are uniquely identified for traceability. British International English spelling and grammar are used throughout.

\subsection{Intended Audience and Reading Suggestions}
This document is intended for project managers, software developers, the quality assurance team, and project stakeholders. 

\subsection{Product Scope}
\begin{cardcontainer}[title={Product Scope}]
    \textbf{CounselorPro} is an intelligent, web-based application designed to provide automated, personalised academic and extracurricular guidance by analysing scanned school term reports. The primary goal is to make high-quality, data-driven academic counselling accessible.

    % REVIEW: Added MVP scope definition based on review feedback.
    \vspace{1em}
    \poppinssemibold\color{AUCDTPrimaryText}Version 1.0 (MVP) Scope:\normalfont The initial release will focus on perfecting the core data pipeline. The scope includes: the user account system, the file upload mechanism, the OCR-based data extraction and user-verification workflow, and a \textbf{rules-based recommendation engine}. The rules-based engine will provide foundational, template-driven advice based on the verified data. More advanced, generative AI features are planned for future versions.
\end{cardcontainer}

\subsection{References}
\begin{itemize}[label={\color{AUCDTBurgundy}\faBook}]
    \item IEEE Std 830-1998, \textit{Recommended Practice for Software Requirements Specifications}.
    \item AUCDT Brand Identity Guide v2.0, August 2025.
\end{itemize}

\section{Overall Description}

\subsection{Product Perspective}
CounselorPro is a new, self-contained, standalone web application. Future versions may include APIs for integration with School Information Systems (SIS).

\subsection{Product Functions}
The major functions are: User Account Management, Report Upload, Data Extraction \& Verification, Performance Analysis, Recommendation Generation, and Report Generation.

\subsection{User Classes and Characteristics}
\begin{cardcontainer}[title={User Classes}]
\begin{description}[\font=\normalfont\poppinsmedium, style=unboxed]
    \item[\poppinssemibold Students:] Aged 10-22. Need a simple, intuitive, and engaging interface.
    \item[\poppinssemibold Parents/Guardians:] Technologically varied. Require a clear overview and value data privacy.
    \item[\poppinssemibold School Counsellors:] Professional users. May use the tool to augment their sessions.
\end{description}
\end{cardcontainer}

\subsection{Operating Environment}
The application will be a web-based platform accessible through modern web browsers on desktop and mobile devices, hosted on a secure cloud infrastructure.

\subsection{Design and Implementation Constraints}
\begin{notebox}
    The system must comply with data protection regulations such as GDPR and FERPA. This is a critical, non-negotiable constraint requiring dedicated resources for security architecture, data handling protocols, and consent management.
\end{notebox}
\begin{itemize}[label={\color{AUCDTBurgundy}\faExclamationTriangle}]
    \item \textbf{C-1 (Data Privacy):} Must comply with GDPR/FERPA.
    \item \textbf{C-2 (OCR):} Must support multiple languages, starting with English.
    \item \textbf{C-3 (AI Ethics):} The recommendation engine must avoid generating biased or deterministic advice. For V1.0, this will be managed via a curated, rules-based system.
    \item \textbf{C-4 (Branding):} The UI must strictly adhere to the AUCDT Brand Identity Guide v2.0.
\end{itemize}

\subsection{Assumptions and Dependencies}
\begin{description}[\font=\normalfont, style=unboxed]
    \item[\poppinssemibold Assumption:] Users have a stable internet connection.
    \item[\poppinssemibold Assumption:] The quality of the uploaded scanned report is sufficient for OCR (e.g. >= 300 DPI). A PoC will be conducted to validate performance on diverse report formats.
    \item[\poppinssemibold Dependency:] The system depends on third-party cloud hosting and AI/OCR APIs. Costs and reliability must be monitored.
\end{description}

\subsection{Non-Functional Requirements}
\subsubsection{Performance}
\begin{itemize}[label={\color{AUCDTCampusGreen}\faTachometerAlt}]
    \item \textbf{NFR-1.1:} Web pages shall load in under 3 seconds.
    \item \textbf{NFR-1.2:} The end-to-end analysis process shall be completed within 90 seconds.
\end{itemize}

\subsubsection{Security}
\begin{itemize}[label={\color{AUCDTCampusGreen}\faShieldAlt}]
    \item \textbf{NFR-2.1:} All user passwords must be hashed using a strong, modern algorithm (e.g., Argon2).
    \item \textbf{NFR-2.2:} All sensitive student data must be encrypted at rest and in transit.
\end{itemize}

\subsubsection{Software Quality Attributes}
\begin{itemize}[label={\color{AUCDTCampusGreen}\faStar}]
    \item \textbf{NFR-3.1 (Reliability):} The system shall have an uptime of 99.5\% or greater.
    \item \textbf{NFR-3.2 (Usability):} The system must comply with WCAG 2.1 Level AA.
\end{itemize}

% REVIEW: Added new NFR category and specific OCR accuracy requirement.
\subsubsection{Data Quality}
\begin{itemize}[label={\color{AUCDTCampusGreen}\faCheckDouble}]
    \item \textbf{NFR-4.1 (OCR Accuracy):} The system shall achieve a Character Error Rate (CER) of less than 5\% on machine-printed text from uploaded reports of at least 300 DPI resolution.
\end{itemize}

% REVIEW: Added new section for Error Handling as recommended.
\subsection{Error Handling and Recovery}
\begin{itemize}[label={\color{AUCDTBurgundy}\faExclamationCircle}]
    \item \textbf{EH-1:} If a user uploads a corrupted or unreadable file, the system shall reject it and display a user-friendly error message explaining the likely problem (e.g., "This file appears to be corrupted. Please try uploading it again.").
    \item \textbf{EH-2:} If the OCR process fails to extract any meaningful data from a file, the system shall notify the user with a message like, "We could not find any readable text in this document. Please ensure it is a school report and try again."
    \item \textbf{EH-3:} The system shall handle API connection failures gracefully, informing the user that the service is temporarily unavailable and suggesting they try again later, without losing their uploaded data if possible.
\end{itemize}

SickBay Management System

The campus clinic's record system: triage, visits, observation beds, daily wellness checks, pharmacy stock, referrals and audit reporting. Database-backed as of July 2026: every record lives in MariaDB on the TUC server, behind WMS single sign-on, and is shared across all signed-in devices.

01Overview

SickBay is a React 19 single-page application served at /sickbay/, backed by an Express server (server.ts run via tsx, PM2 process sickbay, port 3046) and a MariaDB database (tuc_sickbay on port 3306).

The July 2026 migration moved all clinical data out of the browser and into the central database. The SPA now fetches everything from an authenticated REST API and writes back through it; after every write it re-pulls the authoritative dataset, so every device shows the same records. The browser keeps nothing but the display-theme preference.

Access: sign-in is WMS single sign-on with the staff @techbridge.edu.gh Google Workspace account plus a 6-digit authenticator code (TOTP). There are no local passwords, and the API refuses any request without a valid WMS session. Admin features (Pharmacy Stock, Facility Logs, Reports & Sync) additionally require the WMS role SYSTEM_ADMIN, HOD, ADMIN_STAFF or MEDICAL_OFFICER.

03System architecture

The browser talks only to nginx, which proxies the /sickbay/ sub-path to the Express server without stripping the prefix (the server strips it, Pattern 38). Authentication is delegated to WMS: the SPA obtains its session there, and the server re-verifies every request's bearer token against WMS /api/me. Only WMS holds the Google client secret (Pattern 35).

04Database schema

MariaDB database tuc_sickbay on the app-DB instance (localhost:3306), accessed by the scoped user sickbay_app. Seven tables; declared foreign keys link visits, referrals and daily health checks back to patients, and referrals to visits. The visits-to-medications link is enforced by the API's write transaction rather than a constraint.

Entity-relationship diagram, generated from db/migrations/001_init_sickbay.sql and 003_daily_checks_and_med_seed.sql. Standalone copy: erd.svg.

Migrations

FilePurpose

002_seed_staff.sqlSeeds 50 staff patient records, STF001 to STF050. Idempotent (INSERT IGNORE).
003_daily_checks_and_med_seed.sqlAdds the daily_health_checks table, a unique key on medications.batch_number, and seeds the standard 9-line medication inventory. Idempotent; never overwrites adjusted quantities.
Migrations are applied on the server with plesk db < db/migrations/<file>.sql (never a bare mysql client on this Plesk box, Pattern 37). Each file carries its own USE tuc_sickbay; line, so no database flag is needed, and all three are safe to re-run. Database credentials live only in the server's .env, which the deploy script never overwrites.

05API reference

All paths are shown root-relative; in production the SPA calls them under the sub-path (/sickbay/api/...) and the server's strip middleware normalises them. Data routes Auth require Authorization: Bearer <WMS token> and return 401 without one. The three Open routes carry no clinical data.

GET/api/patientsAuthAll patients, ordered by name.
POST/api/patientsAuthCreate a patient; 409 if the patient code already exists.
GET/api/visitsAuthAll visits joined to patients, newest first.
POST/api/visitsAuthTransactional: inserts the visit, deducts dispensed stock, and auto-creates a Pending referral for hospital transfers. Commit or full rollback.
PUT/api/visits/:id/dischargeAuthEnds observation: stamps the end time, appends checkout notes, sets disposition to Back to Class.
GET/api/medicationsAuthInventory, ordered by name.
POST/api/medicationsAuthAdd a stock line.

DELETE/api/medications/:idAuthRemove a stock line.
GET/api/referralsAuthAll referrals joined to patients, newest first.
POST/api/referralsAuthAdd a referral (default status Pending).
PUT/api/referrals/:idAuthUpdate status and outcome notes.
GET/api/facility-logsAuthAll facility fault reports, newest first.

Mapping contract: the database uses snake_case columns; the SPA uses camelCase types. The server owns the mapping in both directions, including JSON-array list columns and the patient identity rule (the SPA's patient.id is the human code such as STF002; the server resolves it to the integer primary key).

06User workflows

Sign in

  1. Open https://ai-tools.techbridge.edu.gh/sickbay/ and click Sign in with your TUC account.
  2. Choose the @techbridge.edu.gh Workspace account (personal Gmail is refused).
  3. Enter the 6-digit authenticator code; first-time users enrol by scanning a QR code.

Log a visit

  1. Click Log New Visit, select the patient, and record vitals, symptoms, severity and treatment.
  2. Dispense medication if needed (allergy conflicts are flagged; out-of-stock lines are blocked).
  3. Choose the disposition. Saving is atomic: visit, stock deduction and any hospital referral are written together.

Everything else

  • Observation beds: patients under observation appear on the dashboard until discharged with checkout notes.
  • Daily checks: the Daily Check form logs routine wellness screenings into the shared journal.
  • Dashboard: Today's Summary (greeting, visits-today trend versus yesterday, critical count, observation count, clinic status) plus urgency-ordered KPI cards led by Critical Alerts.
  • Reports (admin): date-range and daily PDF exports, JSON backup snapshot, searchable audit trail, and a re-sync action that reloads from the database.

07Deployment & verification

Deploys run from the developer machine with .\deploy.ps1 -Build: a local pnpm build, then scp of dist/, docs/, db/, server.ts, src/ (required at runtime by tsx) and the package files, followed by a hard PM2 restart. The server-side .env is created from a template only on the first deploy and never overwritten, so database credentials set once persist.

Definition of done

  • pnpm build completes with no chunk over 600 kB (Pattern 31).
  • pm2 describe sickbay shows uptime in seconds: the new process actually swapped.
  • curl http://localhost:3046/sickbay/api/health returns "db":"connected".
  • An unauthenticated curl to /sickbay/api/patients returns 401 (gate closed), not 500 (database fault) and not 200.
  • A JS asset under /sickbay/assets/ is served as text/javascript, not text/html.

A green deploy banner alone proves nothing; these five checks do.

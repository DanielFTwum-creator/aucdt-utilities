\section{Introduction}

\subsection{Purpose}
This Software Requirements Specification (SRS) document describes the functional and non-functional requirements for the AsanSka University College of Design and Technology Online Application Portal with integrated payment processing via Stripe and PayPal. The system is designed to modernise the application process for prospective students whilst ensuring secure payment processing and compliance with UK data protection regulations.

\subsection{Scope}
The AsanSka University College Application Portal is a web-based platform that facilitates:
\begin{itemize}
    \item Streamlined student registration with UCAS integration capabilities
    \item Dynamic fee calculation based on course type and student residency status (Home/EU vs International)
    \item Secure payment processing through Stripe and PayPal integration
    \item Comprehensive application data collection aligned with UCAS standards for creative and design programmes
    \item Administrative oversight and compliance reporting
    \item Integration with existing Student Information Systems (SIS)
    \item Portfolio upload and assessment tools specific to design and technology programmes
\end{itemize}
The system will benefit prospective students by providing an intuitive application journey tailored to creative disciplines and benefit University College staff by automating administrative processes whilst maintaining rigorous data governance standards.

\section{Overall Description}

\subsection{Product Perspective}
The AsanSka University College Application Portal is a comprehensive web application that integrates with:
\begin{itemize}
    \item Stripe and PayPal payment gateways for secure transaction processing
    \item Email and SMS services for applicant verification and communication
    \item PostgreSQL database with encryption at rest
    \item UCAS Connect API for application data exchange
    \item Existing University College SIS for student record management
    \item Microsoft 365 for email services integration
    \item Creative portfolio management systems for design and technology submissions
    \item Web browsers across all major platforms with accessibility compliance
\end{itemize}

\subsection{Product Functions}
The primary functions include:
\begin{itemize}
    \item Secure applicant registration with enhanced verification processes
    \item Account verification via email and mobile authentication
    \item Dynamic tuition fee calculation based on residency status and programme selection
    \item Multi-gateway payment processing with PCI DSS compliance
    \item Comprehensive application data collection following UCAS standards for creative programmes
    \item Portfolio upload and management system for design and technology applications
    \item Document upload and verification system
    \item Administrative dashboard with comprehensive reporting
    \item Integration with University College's existing student management systems
\end{itemize}

\section{Specific Requirements}

\subsection{Functional Requirements}

\subsubsection{FR-001: User Registration}
\begin{frdescription}
    \item[Priority] High
    \item[Description] The system shall allow prospective applicants to create accounts with minimal required information.
    \item[Inputs] First name, last name, email address, phone number
    \item[Processing] Validate input data, create user account, generate verification token
    \item[Outputs] User account created, verification token sent via email/SMS
\end{frdescription}

\subsubsection{FR-002: Account Verification}
\begin{frdescription}
    \item[Priority] High
    \item[Description] The system shall verify user accounts through token-based verification.
    \item[Inputs] Verification token
    \item[Processing] Validate token, check expiry, activate user account
    \item[Outputs] Account activation confirmation
\end{frdescription}

\subsubsection{FR-003: Tuition Fee Calculation}
\begin{frdescription}
    \item[Priority] High
    \item[Description] The system shall dynamically calculate application and tuition fees based on programme selection and student residency status.
    \item[Inputs] Programme code, student residency status (Home/EU/International), application type
    \item[Processing] Determine fee structure, apply any applicable discounts or scholarships, calculate total fees
    \item[Outputs] Detailed fee breakdown, payment options, estimated total cost
\end{frdescription}

\subsubsection{FR-004: Multi-Gateway Payment Processing}
\begin{frdescription}
    \item[Priority] High
    \item[Description] The system shall integrate with multiple payment providers (Stripe, PayPal) for secure transaction processing.
    \item[Inputs] Payment request, applicant information, fee amount, preferred payment method
    \item[Processing] Route to appropriate payment gateway, process transaction, handle callbacks and webhooks
    \item[Outputs] Payment confirmation, transaction reference, receipt generation
\end{frdescription}

\subsubsection{FR-005: Comprehensive Application Management}
\begin{frdescription}
    \item[Priority] High
    \item[Description] The system shall collect and manage comprehensive application information in accordance with UCAS standards and University College requirements, with special provisions for creative portfolio submissions.
    \item[Inputs] Personal details, educational qualifications, references, personal statement, creative portfolio, supporting documentation
    \item[Processing] Validate application completeness, perform eligibility checks, process creative submissions, store securely with audit trail
    \item[Outputs] Complete application profile, application reference number, status tracking, portfolio review interface
\end{frdescription}

Streamlined registration process requiring only essential information followed by secure account verification.

\begin{itemize}

    \item REQ-4.1.3: System shall send verification tokens via email and SMS

High - Critical for user onboarding

Robust payment processing through multiple gateways with comprehensive financial reporting and compliance.

\subsubsection{Functional Requirements}
\begin{itemize}
    \item REQ-4.2.1: System shall determine fee structure based on programme type and residency status
    \item REQ-4.2.2: System shall provide transparent fee breakdown before payment authorisation
    \item REQ-4.2.3: System shall securely integrate with Stripe for card payments and direct debit
    \item REQ-4.2.4: System shall integrate with PayPal for alternative payment methods
    \item REQ-4.2.5: System shall maintain PCI DSS compliance throughout payment processing
    \item REQ-4.2.6: System shall generate VAT-compliant receipts and invoices
    \item REQ-4.2.7: System shall support instalment payment plans where applicable
\end{itemize}

\subsubsection{Priority}
High - Critical for revenue management and regulatory compliance

Comprehensive application data collection and management capabilities.

\section{Non-Functional Requirements}

\subsection{Performance Requirements}
\begin{itemize}
    \item PERF-001: System shall support 100 concurrent users
    \item PERF-002: Page load times shall not exceed 3 seconds
    \item PERF-003: Payment processing shall complete within 30 seconds
    \item PERF-004: Database queries shall execute within 2 seconds
\end{itemize}

\subsection{Security Requirements}
\begin{itemize}
    \item SEC-001: All data transmission shall use TLS 1.3 encryption as minimum standard
    \item SEC-002: User passwords shall be hashed using bcrypt with appropriate salt rounds
    \item SEC-003: Payment information shall be processed through PCI DSS compliant gateways only
    \item SEC-004: Session management with secure, HTTP-only cookies and CSRF protection
    \item SEC-005: Comprehensive input validation and sanitisation to prevent injection attacks
    \item SEC-006: Implementation of OWASP security guidelines and regular vulnerability assessments
    \item SEC-007: Multi-factor authentication for administrative accounts
    \item SEC-008: Regular security audits and penetration testing
    \item SEC-009: GDPR-compliant data handling with right to erasure capabilities
\end{itemize}

\subsection{Reliability Requirements}
\begin{itemize}
    \item REL-001: System uptime shall be 99.5\% during business hours
    \item REL-002: Automatic backup of critical data every 24 hours
    \item REL-003: Recovery time objective (RTO) of 4 hours
    \item REL-004: Recovery point objective (RPO) of 1 hour
\end{itemize}

\subsection{Availability Requirements}
\begin{itemize}
    \item AVAIL-001: System shall be available 24/7 with planned maintenance windows
    \item AVAIL-002: Graceful degradation during partial system failures
    \item AVAIL-003: Load balancing for high availability
\end{itemize}

\subsection{Maintainability Requirements}
\begin{itemize}
    \item MAINT-001: Modular architecture for easy component updates
    \item MAINT-002: Comprehensive logging for troubleshooting
    \item MAINT-003: Code documentation and commenting standards
    \item MAINT-004: Automated testing coverage minimum 80\%
\end{itemize}

\subsection{Portability Requirements}
\begin{itemize}
    \item PORT-001: Database independence through ORM layer
    \item PORT-002: Cross-platform compatibility
    \item PORT-003: Containerization support for deployment flexibility
\end{itemize}

\subsection{API Endpoints}
\begin{verbatim}
Authentication Endpoints
POST /api/v1/auth/register - Applicant registration
POST /api/v1/auth/verify-email - Email verification
POST /api/v1/auth/verify-mobile - Mobile number verification
POST /api/v1/auth/login - Applicant authentication
POST /api/v1/auth/logout - Session termination
POST /api/v1/auth/password-reset - Password reset functionality

Payment Endpoints
GET /api/v1/payment/fees/{applicantId} - Retrieve applicable fees
POST /api/v1/payment/stripe/intent - Create Stripe payment intent
POST /api/v1/payment/paypal/order - Create PayPal payment order
POST /api/v1/payment/webhooks/stripe - Handle Stripe webhooks
POST /api/v1/payment/webhooks/paypal - Handle PayPal webhooks
GET /api/v1/payment/receipt/{transactionId} - Generate payment receipt

Application Endpoints
POST /api/v1/applications/personal-details - Submit personal information
POST /api/v1/applications/qualifications - Submit educational qualifications
POST /api/v1/applications/references - Submit reference details
POST /api/v1/applications/documents/upload - Document upload functionality
GET /api/v1/applications/status/{applicantId} - Application status tracking
POST /api/v1/applications/submit - Final application submission

Administrative Endpoints
GET /api/v1/admin/applications - List all applications (paginated)
GET /api/v1/admin/reports/financial - Financial reporting
PUT /api/v1/admin/applications/{id}/status - Update application status
\end{verbatim}
